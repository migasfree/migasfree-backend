# Service: Mutual TLS & Secure Cryptography

> **Scope:** Web Server Layer (Nginx/HAProxy) & Django Middleware
> **Access Privilege:** **mTLS Enrolled Clients**
> **Module:** `migasfree.core`

---

## 1. Overview

`migasfree-backend` employs a double-layer security architecture to secure all administrative actions. Communication on restricted endpoints requires Mutual TLS (**mTLS**) authentication at the web server layer, backed by custom cryptographic signature verification at the Django application layer.

---

## 2. Double-Layer Security Flow

The secure handshaking follows a strict two-tier verification:

```txt
[Client Agent]
      │
      │ 1. Mutual TLS Handshake (Client Cert & Private Key)
      ▼
┌────────────────────────────────────────────────────────┐
│      Nginx / HAProxy Reverse Proxy Layer               │
│      - Verifies Client Cert against Private CA         │
│      - Extracts Client Cert fields (UUID, Name)        │
│      - injects headers: 'HTTP_X_CLIENT_CERT'           │
└─────────────────────────┬──────────────────────────────┘
                          │
                          │ 2. Proxy request with Cert Headers
                          ▼
┌────────────────────────────────────────────────────────┐
│      Django Application Layer (Middleware)             │
│      - Parses Certificate Headers                      │
│      - Validates UUID against registered computers     │
│      - Decrypts and verifies JWS payload signature      │
│        using the stored project private key            │
└────────────────────────────────────────────────────────┘
```

---

## 3. Cryptographic Key Enrollment & Project Security

1. During `register`, the client posts to `/api/v1/public/keys/project/` using project credentials.
2. The server generates asymmetric key pairs:
   - `server.pub` is sent to the client to verify server signatures.
   - `<project>.pri` is generated to verify client-signed requests.
3. The server signs client configuration files using the private project key, ensuring clients only parse configurations generated by an authorized server.
4. **JWS Signature Verification**: All post requests to `/api/v1/safe/` must wrap their payload data in JSON Web Signature (**JWS**) format. Custom middlewares in the backend intercept these requests, verify the signature using the stored project public key, and parse the raw JSON data to downstream DRF serializers.
